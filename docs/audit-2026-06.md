# GrenMet Health & Risk Audit — June 2026

> **Historical record (point-in-time).** This audit predates the 2026-06 consolidation. The standalone `cap`/`hr`/`wxwatch`/`wxproducts`/`salesbus` apps were folded into `admin-gms` (their `apps/web/<app>/` dirs deleted), so the `file:line` citations below reflect the structure at audit time and may no longer resolve — code referenced as `apps/web/cap/...` / `apps/web/wxwatch/...` now lives under `apps/web/admin-gms/`. Findings are preserved as written; verify against current code before acting.

**Scope:** Health & risk audit producing a prioritized, actionable findings list.
**Dimensions (core):** code quality/consistency, security, testing, API-contract drift. Secondary dimensions (deps, CI, design-system, docs) only where a core finding leads there.
**Coverage:** tiered — T1 deep (shared auth, FastAPI, wxproducts, wxwatch, cap), T2 standard, T3 light.
**Method:** tooling-first evidence + manual review. Findings = locate + root-cause + recommended fix.
**Ranking:** Severity (Critical/High/Medium/Low) × effort (S/M/L).

> Out of scope: the vendored `capcomposer` Wagtail/Django app (git submodule under `apps/api/fastapi/src/cap/`); already-tracked workstreams (design-system migration, CI/CD backlog) are referenced, not re-reported.

## Severity rubric

- **Critical** — data loss, auth bypass, secret exposure, broken public API contract
- **High** — user-facing breakage, security weakness needing a chained condition, type-unsafety in core paths
- **Medium** — correctness/consistency risk contained to one app, missing tests on existing logic
- **Low** — style/convention drift, cosmetic inconsistency

## Tier 1A — Shared auth package + FastAPI core

No Critical findings. One High.

| # | Sev | Eff | Finding | Location |
|---|-----|-----|---------|----------|
| 1 | **High** | S | `SECRET_KEY` defaults to `secrets.token_urlsafe(32)`. A *missing* env var in staging/prod passes every validator (only literal `"changethis"` + length/charset are checked), silently yielding a per-process ephemeral key → cross-worker token rejection (intermittent 401s) + all tokens invalidated on every restart. The guard gives false confidence. **Fix:** no default in non-local; require explicit `SECRET_KEY` or fail validation. | `src/auth/config.py:20` |
| 2 | **Medium** | S | `NewPassword.new_password` / `UpdatePassword.new_password` carry **no length constraint**, while signup/login enforce `min_length=8, max_length=40`. Password reset/change accept a 1-char password, bypassing the minimum enforced elsewhere. **Fix:** apply `Field(min_length=8, max_length=40)`. | `src/auth/schemas.py:48,201` |
| 3 | **Medium** | S | `ACCESS_TOKEN_EXPIRE_MINUTES = 8 days`. The OAuth2 password-grant JWT is long-lived **and non-revocable** (logout revokes sessions, not bearer tokens). The web/session flow is fine (15-min token + revocable opaque session). **Fix:** shorten the legacy token or scope it machine-only. | `src/auth/config.py:22` |
| 4 | **Medium** | M | `_check_default_secret` + `_validate_secret_strength` are copy-pasted verbatim in `Settings` and `AuthConfig`; two `BaseSettings` independently parse `.env.local` and redefine `ENVIRONMENT`. Drift risk (finding #1 lives in this gap). **Fix:** extract a shared mixin/util. | `src/config.py:74-136`, `src/auth/config.py:29-66` |
| 5 | **Medium** | S | CORS uses `allow_credentials=True` with `allow_origins=settings.all_cors_origins` and no guard rejecting `*`/over-broad origins. **Fix:** assert no wildcard; document required origins. | `src/main.py:123-130` |
| 6 | Low | S | JWT carries only `exp`+`sub` — no `iat`/`nbf`/`iss`/`aud`. (Alg pinned to HS256 — good.) Hardening only. | `src/auth/utils.py:26-31` |
| 7 | Low | S | `/login/session/access-token`, `/refresh`, `/logout-all` are unthrottled, unlike rate-limited login. Secrets are high-entropy (not brute-forceable), but exchange runs every SSR page load — DoS amplifier. | `src/auth/routers/login.py:210,247,316` |
| 8 | Low | S | Public CAP `/{identifier}.xml` relies on the invariant "snapshots only exist post-publish" rather than an explicit lifecycle filter (siblings all filter). Currently safe; fragile. **Fix:** add the lifecycle join. | `src/cap/service.py:719` |
| 9 | Low | S | `/scalar` docs endpoint registered unconditionally — serves a broken reference in prod (`openapi_url` is None there). Harmless. | `src/main.py:173` |

### Verified-solid (positives)
Session secrets 48-byte `token_urlsafe`, **SHA-256 hashed at rest**; JWT **alg pinned** (no alg-confusion), `sub` UUID-validated, inactive users rejected; login + recovery **rate-limited**; password recovery **non-enumerating**; **docs hidden in prod**; `response_model` enforced (no DB-field leakage); public CAP list/detail endpoints **correctly filter to published states**; signup/login password capped at 40 (bcrypt 72-byte truncation moot).

## Tier 1B — Data-layer apps (wxproducts, wxwatch, cap web) + API test coverage

No Critical findings. The data-layer apps are largely clean; the cap admin section is unfinished scaffolding.

| # | Sev | Eff | Finding | Location |
|---|-----|-----|---------|----------|
| 10 | **Medium** | S | cap admin dashboard `getAdminAlerts()` fetches the protected `/api/v1/cap/alerts` (FastAPI, via `CAP_API_URL`) **server-side with no Authorization header**, then swallows the resulting 401 (`if (!response.ok) return empty`). The dashboard renders "No CAP alerts are available" for everyone — it can never show real data. The authenticated proxy at `api/[...path]/route.ts` (which attaches the Bearer token) exists but isn't used here. **Fix:** route the server fetch through the authenticated proxy, or replicate `readSessionCookie → exchangeSessionForAccessToken → Bearer`. | `apps/web/cap/src/app/admin/page.tsx:107-117` |
| 11 | **Medium** | M | cap `/admin/*` gate (`proxy.ts`, Next 16 middleware) checks only **cookie presence, not validity** — any non-empty `grenmet_session` value passes and loads the admin shell. Data stays protected (FastAPI requires a valid token), so it's UI exposure, not data exposure. Must be hardened before the admin section goes live. **Fix:** validate the session (exchange) at the page/layout level in addition to the presence check. | `apps/web/cap/src/proxy.ts:24-27` |
| 12 | **Medium** | M | `wxwatch.getImagesGroupedByName()` does `db.select().from(weatherImages)` with **no limit** and groups in memory on every gallery load. The scraper appends images indefinitely → unbounded full-table scan + memory growth over time. (`getImagesByDateAndSynoptic` is correctly day-bounded.) **Fix:** bound by recency or aggregate in SQL. | `apps/web/wxwatch/src/db/queries.ts:29-52` |
| 13 | **Medium** | M | Web apps (cap, wxwatch, wxproducts) appear to carry **no unit/integration tests**, despite the repo's tests-alongside-features mandate. Logic-bearing code is untested — e.g. wxwatch synoptic-time grouping, cap adapters. (FastAPI, by contrast, has strong coverage — see positives.) **Fix:** backfill tests for the logic-bearing modules; enforce in CI. *(Verify: confirm no `*.test.ts` exist in these apps.)* | `apps/web/{cap,wxwatch,wxproducts}` |
| 14 | Low | S | cap `admin/new/page.tsx` is a non-functional UI stub — "Save Draft" buttons have no submit handler; the form posts nowhere. Informational: the admin section is incomplete. | `apps/web/cap/src/app/admin/new/page.tsx` |
| 15 | Low | S | Health endpoints (`wxwatch`, `wxproducts`, cap) return `{status:"ok"}` **without checking DB/upstream connectivity** → false-green to orchestrators when a dependency is down. | `apps/web/*/src/app/api/health/route.ts` |
| 16 | Low | S | `test_create_permission` and `test_create_role` assert HTTP `200`, but the create routes correctly declare `status_code=201`. These tests **currently fail** (`assert 201 == 200`) — CI is either red or not running them. Routes are correct; the assertions are stale. **Fix:** update asserts to `201`. *(Found while running the suite during remediation.)* | `tests/auth/routers/test_permissions.py:19`, `test_roles.py:19` |

### Verified-solid (positives)
- **No raw SQL, no `"use server"` actions** in wxproducts/wxwatch — Drizzle parameterizes everything (no SQL-injection surface; no unauthenticated mutation entry point).
- `weather_images` has no per-user rows → no row-scoping/IDOR concern; wxwatch auth gates app access via `redirect` + `proxy.ts`.
- wxwatch logout route correctly follows the documented pattern (`readSessionCookie` → `logoutSession` in try/catch → `clearSessionCookieOnResponse`).
- cap's authenticated API proxy (`api/[...path]/route.ts`) is well-built: strips hop-by-hop headers, attaches Bearer, clears cookie on upstream 401, returns 503 when the auth service is unreachable.
- **FastAPI test coverage is strong**: dedicated `tests/auth/` (login, permissions, roles, users, service, utils, models), `tests/cap/` (api, state machine, XML/geo validation), `tests/hr/` (authz, policy, dependencies, leave ledger, workflow, roster). The security-critical authz paths are tested.

### Tier-1 dimension summary
- **Code quality/consistency:** clean at tooling level (type-check + Biome green); module-import & two-layer-model conventions consistently followed in FastAPI.
- **Security:** one High (#1 SECRET_KEY); the rest are Medium config-hardening + forward-looking admin-gate items. No auth bypass or data leak found.
- **Testing:** strong on the API, a gap on the web apps (#13).
- **API-contract drift:** `gen/` in sync; live-vs-committed openapi regeneration still recommended as a CI guard.
