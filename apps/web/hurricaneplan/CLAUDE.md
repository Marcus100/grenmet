# hurricaneplan (`@grenmet/web-hurricaneplan`) — Claude context

Port **3003**. Hurricane preparedness plan — a static MDX documentation site for airport meteorological operations.

## Turbopack

Runs Turbopack. Config file is `next.config.ts`. Do not add `--webpack`.

```bash
pnpm dev    # runs: node scripts/generate-sections.mjs && next dev --turbopack --port 3003
pnpm build  # runs: node scripts/generate-sections.mjs && next build --turbopack
```

## Auth pattern

Delegates to `web-auth` (`:3000`) via redirect — does not handle sign-in itself.

## MDX pipeline — Content Collections

MDX is compiled outside the Turbopack pipeline via `@content-collections/next`. This is how custom remark/rehype/recma plugins (which are functions) can be used with Turbopack — they run in a separate Node.js process, not as loader options.

| File | Purpose |
|---|---|
| `content-collections.ts` | Collection definition — schema, plugins, transform |
| `src/content/**/*.mdx` | All content pages (YAML frontmatter for title/description) |
| `src/app/[[...slug]]/page.tsx` | Catch-all route — renders any content page |
| `src/components/MdxContent.tsx` | MDX renderer using `MDXContent` from `@content-collections/mdx/react` |
| `src/mdx/search.ts` | FlexSearch module — imports pre-built `src/data/sections.json` |

Plugins run inside `content-collections.ts` transform:
- remark: `remark-gfm`, `mdx-annotations`, custom `captureH2s` (extracts h2 sections)
- rehype: `mdx-annotations`, custom `rehypeParseCodeBlocks`, `rehypeShikiHighlight`, `rehypeSlugify`
- recma: `mdx-annotations`

Syntax highlighting via Shiki (`shiki@0.14`, old API — matches what `Code.tsx` expects). Do not add Prism or other highlighters. Do not upgrade to Shiki v1 without updating `Code.tsx`.

## MDX component mapping

`MdxContent.tsx` passes all components to `MDXContent`. Do not add bare `import` statements inside `.mdx` files — add the component to `MdxContent.tsx`'s `components` map instead.

Global MDX components (available on every page): `Button`, `Note`, `CodeGroup`, `code`, `pre`, `h2`, `a`, `wrapper`, `Row`, `Col`, `Properties`, `Property`

Page-specific components (also wired in globally): `Guides`, `HeroPattern`, `Resources`, `Libraries`

## Search index

`scripts/generate-sections.mjs` scans `src/content/**/*.mdx` and writes `src/data/sections.json` (h1 page titles + h2 section slugs). Runs automatically before `next dev`/`next build`. Re-run manually if content changes during dev:

```bash
node scripts/generate-sections.mjs
```

`src/mdx/search.ts` imports `sections.json` at module load and builds the FlexSearch client-side index. `Search.tsx` lazy-imports it when the search dialog opens.

## Sections / sidebar TOC

`layout.tsx` imports `allHurricanepages` from `content-collections` and builds `allSections: Record<string, Section[]>` from each page's `sections` field (extracted by `captureH2s` during compilation). The sidebar TOC and scroll-spy both depend on h2 IDs — these are generated by `rehypeSlugify` (same slugifier, same order as `captureH2s`).

## Key dependencies (unique to this app)

- `@content-collections/core` + `@content-collections/mdx` + `@content-collections/next` — MDX pipeline
- `mdx-annotations` — `{{ className: 'lead' }}` annotation syntax in MDX
- `@algolia/autocomplete-core` — search autocomplete
- `flexsearch` — client-side full-text search
- `motion` — page transitions and animations (import from `motion/react`)
- `zustand` — state management
- `@tailwindcss/typography` — prose styling for MDX content
- `shiki@0.14` — syntax highlighting (old API)

## Adding content pages

1. Create `src/content/your-slug.mdx` with YAML frontmatter:
   ```
   ---
   title: 'Page Title'
   description: 'Optional description.'
   ---
   ```
2. Add the route to `src/components/Navigation.tsx` if it should appear in the sidebar.
3. Re-run `node scripts/generate-sections.mjs` if dev server is already running.
